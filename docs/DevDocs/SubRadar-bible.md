# Subscription Tracker — Библия проекта

> Документ фиксирует ключевые архитектурные решения, технологический стек и принципы разработки приложения для трекинга подписок.

---

## 1. Концепция продукта

Мобильное приложение для отслеживания активных подписок пользователя. Приложение должно быть **open-source**, поддерживать как локальное хранение данных, так и синхронизацию через сервер.

### Три режима работы

| Режим | Описание | Аудитория |
|---|---|---|
| **Локальный** | Данные хранятся только на устройстве | Пользователи, не доверяющие облаку |
| **Централизованный сервер** | Синхронизация через общий платный сервер | Платные пользователи (будущая монетизация) |
| **Self-hosted сервер** | Пользователь разворачивает бэкенд на своём сервере | Технически продвинутые пользователи |

При первом запуске приложения пользователь выбирает один из трёх режимов.

## 2. Технологический стек

### iOS клиент
- **Язык:** Swift
- **UI фреймворк:** SwiftUI
- **Локальное хранилище:** SwiftData (iOS 17+) или GRDB (SQLite) для поддержки старых версий iOS

### Бэкенд
- **Язык:** Go (Golang)
- **HTTP роутер:** Chi (лёгкий, идиоматичный) или Gin
- **Hot reload при разработке:** Air

### База данных
- **Self-hosted:** SQLite по умолчанию (нет внешних зависимостей, один файл)
- **Централизованный сервер:** PostgreSQL
- **Работа с SQL:** sqlc — генерация типобезопасного Go-кода из SQL-запросов
- **Миграции:** golang-migrate

### Синхронизация
- Архитектура **offline-first**
- Инкрементальная синхронизация на основе timestamp или offset
- Клиент может работать без сети, данные синхронизируются при подключении

---

## 3. Почему Go для бэкенда

- Компилируется в **единый статический бинарник** — идеально для self-hosted
- Минимальное потребление ресурсов (работает даже на Raspberry Pi)
- Кросс-компиляция под все платформы одной командой
- Простой язык — низкий порог входа для контрибьюторов
- Быстрая разработка по сравнению с Rust при сопоставимой производительности для REST API

## 4. Дистрибуция и деплой

### Принцип: без Docker

Docker упрощает деплой, но создаёт накладные расходы. Go позволяет обойтись без него благодаря статическим бинарникам.

### GitHub Releases

Основной способ дистрибуции. При каждом релизе GitHub Actions автоматически собирает бинарники:

```
Релиз v1.0.0
├── subtracker-linux-amd64
├── subtracker-linux-arm64        # Raspberry Pi, ARM серверы
├── subtracker-darwin-arm64       # Mac M1/M2/M3
├── subtracker-darwin-amd64       # Mac Intel
├── subtracker-windows-amd64.exe
└── checksums.txt                 # SHA256 для верификации
```

### Пакетные менеджеры (в приоритете)
- **Homebrew** (macOS/Linux) — `brew install subtracker`
- **Scoop** (Windows)
- **deb-пакет** — для Debian/Ubuntu

### Systemd (Linux self-hosted)

К документации прикладывается готовый `.service` файл для автозапуска бэкенда через systemd.

## 5. Структура репозитория

Монорепозиторий — весь код в одном GitHub репозитории. Упрощает issues, PR и документацию для контрибьюторов.

```
subscription-tracker/
│
├── ios/                            # Swift приложение
│   ├── SubscriptionTracker/
│   └── SubscriptionTracker.xcodeproj
│
├── backend/                        # Go сервер
│   ├── cmd/
│   │   └── server/
│   │       └── main.go             # Точка входа
│   ├── internal/                   # Внутренняя логика (не экспортируется)
│   │   ├── handlers/
│   │   ├── models/
│   │   └── storage/
│   ├── migrations/                 # SQL миграции
│   └── go.mod
│
├── docs/                           # Документация
│   ├── api.md                      # Описание API эндпоинтов
│   ├── self-hosting.md             # Гайд по self-hosted установке
│   └── contributing.md
│
├── design/                         # Figma экспорт, иконки, assets
│
├── .github/
│   └── workflows/
│       └── release.yml             # CI/CD для автосборки бинарников
│
├── README.md
└── LICENSE                         # MIT лицензия
```

### Ветки

| Ветка | Назначение |
|---|---|
| `main` | Стабильный код, только через PR |
| `dev` | Тестирование и мердж изменений |
| `backend` | фичи бэкэнд |
| `ios` | фичи ios приложения |

---

## 6. Процесс разработки

### Порядок работы

1. Разработка фич для ios или back в отдельных ветках
2. PR изменений из соответствующих веток в dev
3. Тестирование работы в dev
4. Мердж dev в main
5. Выпуск необходимых тегов
